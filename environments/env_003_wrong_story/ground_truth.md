# Ground truth — env_003_wrong_story

This network has been compromised. The intrusion below is the planted incident the Hunter is meant to uncover — it is the **same incident** as in `env_000_dummy`, `env_001_noisy`, and `env_002_competing_narratives`, unchanged in timeline, techniques, and indicators. This environment carries forward env_001's eight `decoy_*` benign lookalikes and env_002's two `competing_*` narratives, and adds two new wrinkles: (1) a third, much **louder** competing narrative — an authorized red-team/assumed-breach exercise running concurrently at the branch office, which on its own reads as a second, more obvious intrusion; and (2) a genuine **evidence gap** — PowerShell Script Block Logging is disabled on dc01, so the malicious scheduled task's payload cannot be pulled via EID 4104 and must be recovered through other telemetry. The "current date" for this environment is 2026-06-12 (see `topology.json`) — all timeline events fall in the few days prior, well within a "last 7 days" or "last 2 weeks" query window.

## Overview

A phishing email opened on FIN-ALICE (ws01) led to a backdoor, credential theft, lateral movement to the domain controller (dc01), persistence on the DC, and exfiltration of Finance documents from the file server (fs01) — all over the course of a few hours overnight on 2026-06-09, preceded by initial access on 2026-06-08. This is the real, ground-truth incident and should be reported in full, exactly as in env_000/001/002.

This environment carries forward the env_001 decoys and the env_002 competing narratives (a second host's EDR-validated LSASS access, and an unrelated external credential-stuffing burst), and adds a third: in the same overnight window, the branch office (bws01) shows what looks like an active, EDR-alerting intrusion of its own — a beaconing implant, a Mimikatz credential dump, a persistence task, and an RDP pivot into srv02 — that is in fact an authorized assumed-breach red-team exercise (ticket SEC-2026-0085), not part of the real attack. A good investigation should surface the real incident in full, correctly set aside the env_001/002 lookalikes, and correctly resolve this third narrative — recognizing that "louder and more alarming" does not mean "more real." This environment also includes a genuine evidence gap (PowerShell Script Block Logging disabled on dc01) that the investigation must work around rather than treat as suspicious in itself.

## Timeline (unchanged from env_000/001/002)

**2026-06-08, ~14:32 (Mon, business hours)** — *Initial access*
Alice Chen (achen) receives an email with subject "Invoice #84512 — Payment Overdue" from a spoofed vendor address, with attachment `Invoice_84512.xlsm`. She opens it and enables macros. The macro drops `winupdate32.exe` into `C:\Users\achen\AppData\Roaming\Microsoft\Windows\winupdate32.exe` and adds a Run-key persistence entry `HKCU\Software\Microsoft\Windows\CurrentVersion\Run\WindowsUpdateHelper` pointing at it. The backdoor begins beaconing to `185.220.101.47` over HTTPS (443), roughly every 5 minutes.

**2026-06-09, ~02:14 (Tue, off-hours)** — *Credential access on ws01*
Via the backdoor, the attacker briefly spawns a process (`svchost_helper.exe`) that accesses `lsass.exe` on ws01 via the `comsvcs.dll` MiniDump technique (invoked through `rundll32.exe`) and dumps credential material. Among the recovered credentials are cached domain credentials for `CORP\bnguyen` (Bob Nguyen, IT admin) — Bob had RDP'd into ws01 on 2026-06-05 for a support ticket, leaving his credentials cached on the machine.

**2026-06-09, ~02:31 (Tue, off-hours)** — *Lateral movement to dc01*
Using `CORP\bnguyen`'s credentials, the attacker opens an RDP session (logon type 10, RemoteInteractive) from ws01 (10.10.10.21) to dc01 (10.10.0.10). This is anomalous on two counts: it's ~5.5 hours outside Bob's normal working hours, and it originates from Alice's workstation rather than Bob's own (IT-BOB, 10.10.10.35) — Bob is not logged into anything at this time.

**2026-06-09, ~02:40 (Tue, off-hours)** — *Persistence on dc01*
From the dc01 session, the attacker:
- Creates a scheduled task named `MicrosoftEdgeUpdateTaskMachine`, running `powershell.exe -enc <base64>` daily at 03:00, which re-establishes a connection to `185.220.101.47`.
- Creates a new local account `svc_backup` on dc01 and adds it to the local Administrators group, as a secondary persistence mechanism.

**2026-06-09, ~02:55–03:10 (Tue, off-hours)** — *Exfiltration via fs01*
Still using `CORP\bnguyen`'s session on dc01, the attacker connects via SMB to `\\CORP-FS01\Finance\Confidential` (fs01, 10.10.0.20) and copies several files — Q2 financial statements, a payroll spreadsheet, and draft M&A documents — to a staging folder `C:\Windows\Temp\sysdiag\` on dc01. The staged files are compressed into `sysdiag_2026-06-09.zip`, which is then uploaded to `185.220.101.47` via the scheduled task's outbound connection at ~03:10.

**2026-06-10 onward** — *Dormancy*
The ws01 backdoor continues its ~5-minute beacon to `185.220.101.47`, low-and-slow, blending into normal outbound web traffic. The dc01 scheduled task fires daily at 03:00 as configured. No further escalation has occurred as of the current date (2026-06-12).

## Evidence gaps (NEW in env_003)

Unlike the decoys and competing narratives above (which are extra activity to sift through), this is a genuine **absence** of evidence that the investigation has to work around. Don't manufacture data to fill this gap — return `found: false` with the note described.

| Tag | What's missing | Details |
|---|---|---|
| `evidence_gap_dc01_4104` | PowerShell Script Block Logging (EID 4104) on dc01 | The Group Policy enabling `Microsoft-Windows-PowerShell/Operational` Script Block Logging (`EnableScriptBlockLogging`) has never been applied to dc01 or any other server-class host in corp-lan. The operational log channel exists but contains no EID 4104 entries for the period in question. A request specifically for EID 4104 on dc01 — or for "the decoded/full PowerShell command via script block logging" — should return `found: false`, with a note that Script Block Logging is not enabled on this host (e.g. the relevant registry key/GPO setting is absent, confirmable via `gpresult` or the registry). The full `-enc <base64>` command line for the malicious `MicrosoftEdgeUpdateTaskMachine` task remains recoverable through other channels that **should** return `found: true`: the Task Scheduler operational log (EID 4698, which includes the task's full action/argument XML at creation time) and Sysmon EID 1 (ProcessCreate, which logs the full command line for `powershell.exe` regardless of script-block-logging state) — both of these were already demonstrated as available in env_002 and should remain so here. |
